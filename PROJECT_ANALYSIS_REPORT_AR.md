# تقرير تحليل مشروع Mid Aid Backend

## 1) الملخص التنفيذي
المشروع مبني على `NestJS + TypeORM` وبنية الموديولات واضحة ومناسبة للتوسع. لكن توجد مشاكل عالية الأثر في الأمان، صلاحيات الوصول، اتساق البيانات، وجهوزية الإنتاج.  
أهم المخاطر الحالية:
- ثغرات صلاحيات في بعض مسارات الطلبات والمستخدمين قد تسمح بالوصول لبيانات لا تخص المستخدم.
- إعدادات تشغيل غير آمنة للإنتاج (`synchronize: true` + وجود fallback ثابت لسر JWT).
- منطق السلة والطلبات لا يمنع تجاوز المخزون بشكل موثوق، ولا يخفض المخزون عند إنشاء الطلب.
- ضعف كبير في الاختبارات (اختبارات هيكلية فقط) مع عدم القدرة على التحقق البنائي حاليًا بسبب بيئة اعتماديات غير مكتملة.

## 2) طريقة التحليل
- مراجعة هيكل المشروع وملفات الضبط الأساسية.
- مراجعة الخدمات الحرجة: `auth`, `user`, `order`, `cart`, `pharmacy`, `emergencyOrder`, `upload`, `mail`.
- مراجعة DTO والتحقق من المدخلات.
- محاولة تشغيل `build` و`test` للتحقق التشغيلي.

نتيجة التشغيل في البيئة الحالية:
- `pnpm run build` و `pnpm run test` فشلا بسبب غياب ملفات تنفيذ في `node_modules` (مثل `@nestjs/cli` و`jest`).
- محاولة إعادة التثبيت فشلت بسبب أخطاء وصول للشبكة/السجل (`EACCES` على `registry.npmjs.org`).

## 3) المشاكل والتحديات (مرتبة حسب الأولوية)

### A) مشاكل حرجة (Critical)
1. إعداد قاعدة البيانات غير آمن للإنتاج
- الدليل: `src/app.module.ts:62` (`synchronize: true`).
- الأثر: تعديل المخطط تلقائيًا في الإنتاج قد يسبب فقد بيانات أو سلوك غير متوقع.
- المقترح:
  - جعلها `false` افتراضيًا.
  - الاعتماد على migrations فقط.
  - إنشاء pipeline واضح لتطبيق migrations قبل الإقلاع.

2. وجود fallback سري ثابت لتوقيع JWT
- الدليل: `src/modules/auth/jwt.strategy.ts:17`, `src/modules/auth/auth.module.ts:18`.
- الأثر: إذا غاب `JWT_SECRET` فسيعمل النظام بسر افتراضي معروف.
- المقترح:
  - إزالة fallback تمامًا.
  - الإقلاع يجب أن يفشل إذا لم يوجد `JWT_SECRET`.

3. ثغرة صلاحيات في تحديث حالة الطلب
- الدليل: `src/modules/order/services/order.service.ts:177`.
- المشكلة: لا يوجد تحقق أن الصيدلي يخص نفس الصيدلية في الطلب، ولا تحقق صريح أن المستخدم يملك الطلب قبل الإلغاء.
- الأثر: مستخدم/صيدلي قد يعدل حالة طلب لا يخصه (حسب الدور).
- المقترح:
  - إضافة تفويض على مستوى المورد:
    - `USER`: فقط طلباته.
    - `PHARMACIST`: فقط طلبات صيدليته.
    - `ADMIN`: كل الطلبات.

4. كشف تفاصيل الطلب دون تحقق ملكية/صلاحية
- الدليل: `src/modules/order/controllers/order.controller.ts:68` و `src/modules/order/services/order.service.ts` (`findOrderDetails`).
- الأثر: أي مستخدم مصادق يمكنه رؤية بيانات طلبات الآخرين إذا عرف `orderId`.
- المقترح:
  - تمرير المستخدم الحالي للخدمة.
  - فرض نفس قواعد التفويض السابقة على القراءة.

### B) مشاكل عالية (High)
5. مسارات المستخدمين مكشوفة لكل مستخدم مصادق
- الدليل: `src/modules/user/user.controller.ts:51`, `src/modules/user/user.controller.ts:61`.
- المشكلة: `GET /users` و `GET /users/:id` محميان JWT فقط بدون دور `ADMIN`.
- الأثر: تسريب بيانات مستخدمين.
- المقترح:
  - إضافة `RolesGuard + Roles(UserRole.ADMIN)` لمسارات الإدارة.
  - توفير endpoint منفصل لملف المستخدم الحالي (`/users/me`) فقط.

6. مسار إنشاء Admin علني مع secret عبر body
- الدليل: `src/modules/user/user.controller.ts:38`, `src/modules/user/user.service.ts:62`.
- الأثر: يزيد سطح الهجوم (محاولات brute-force على `SECRET_KEY_ADMIN`).
- المقترح:
  - إغلاق المسار في الإنتاج أو تقييده بشبكة داخلية/إعداد bootstrap إداري مرة واحدة.
  - على الأقل: rate-limit مشدد + تدقيق/تسجيل أمني + تعطيل بعد أول admin.

7. منطق السلة يسمح بتجاوز المخزون
- الدليل: `src/modules/cart/cart.service.ts:47`.
- المشكلة: عند وجود العنصر مسبقًا يزيد الكمية بدون إعادة مقارنة الإجمالي بالمخزون.
- الأثر: يمكن أن تصبح كمية السلة أعلى من المتاح.
- المقترح:
  - قبل الحفظ: التحقق من `(existing.quantity + dto.quantity) <= stock.quantity`.

8. تعديل/حذف عنصر السلة حسب `productId` فقط
- الدليل: `src/modules/cart/cart.service.ts:68`, `src/modules/cart/cart.service.ts:85`.
- المشكلة: إذا كان نفس المنتج من أكثر من صيدلية، يتم استهداف أول تطابق فقط.
- الأثر: سلوك خاطئ وعدم حتمية.
- المقترح:
  - جعل المفتاح `productId + pharmacyId` أو استخدام `cartItemId`.

9. إنشاء الطلب لا يخصم المخزون
- الدليل: `src/modules/order/services/order.service.ts:37` (منطق إنشاء الطلب).
- الأثر: overselling عند تعدد الطلبات.
- المقترح:
  - خصم المخزون داخل نفس transaction.
  - رفض الطلب إذا المخزون تغير قبل الإتمام (optimistic/pessimistic locking).

10. رفع الملفات بلا قيود حجم/نوع على مستوى interceptor
- الدليل: `src/modules/upload/upload.controller.ts:25`.
- الأثر: استهلاك ذاكرة/DoS أو رفع ملفات غير مرغوبة قبل فلترة Cloudinary.
- المقترح:
  - تفعيل `limits.fileSize` و `fileFilter` في `FileInterceptor`.
  - إضافة مصادقة/تفويض لمسار الرفع.

### C) مشاكل متوسطة (Medium)
11. خطأ منطقي في التحقق من الإحداثيات (0 يعتبر غير صالح)
- الدليل: `src/modules/emergencyOrder/services/emergencyOrder.service.ts:31`.
- المشكلة: `if (!latitude || !longitude)` يرفض قيمة `0`.
- المقترح:
  - استخدام `latitude == null || longitude == null`.

12. مخرجات عربية/رموز بها تشوه ترميز (Mojibake)
- الدليل: `src/common/logger/custom-logger.ts:5`.
- الأثر: سجلات ورسائل غير مفهومة.
- المقترح:
  - توحيد ترميز الملفات إلى UTF-8.
  - مراجعة السلاسل العربية المعطوبة في الكود.

13. تكرار مسارات Legacy يزيد التعقيد
- الدليل: مثال `src/modules/pharmacy/pharmacy.controller.ts` (عدة aliases).
- الأثر: صيانة أصعب واحتمال تضارب سلوك.
- المقترح:
  - خطة deprecation بإصدارين:
    - إصدار 1: الإبقاء مع تحذير.
    - إصدار 2: إزالة المسارات القديمة.

14. ازدواجية متغيرات البريد في البيئة
- الدليل: `.env.example:11-20`.
- المشكلة: وجود `EMAIL_*` و `SMTP_*` بينما التطبيق يعتمد `SMTP_*`.
- المقترح:
  - إزالة غير المستخدم وتوحيد التسمية.

15. لا يوجد نظام migrations فعلي رغم وجود datasource
- الدليل: `src/typeorm.config.ts` يحدد مجلد migrations، لكن لا يوجد `src/migrations`.
- المقترح:
  - إضافة أول migration baseline.
  - اعتماد سياسة schema changes عبر migrations فقط.

### D) فجوات جودة (Quality Gaps)
16. الاختبارات الحالية ضعيفة جدًا
- الدليل: `src/modules/user/user.service.spec.ts` و `src/modules/user/user.controller.spec.ts` (فقط `should be defined`).
- الأثر: أعطال منطق الأعمال تمر للإنتاج دون كشف.
- المقترح:
  - اختبارات وحدات لخدمات `order/cart/auth/pharmacy`.
  - اختبارات تكامل لمسارات الصلاحيات.
  - e2e لسيناريوهات حرجة (checkout, status transitions, emergency flow).

17. لا يمكن التحقق البنائي حاليا بسبب الاعتماديات
- الدليل: فشل build/test لغياب bin modules ثم فشل `pnpm install` بـ `EACCES`.
- المقترح:
  - إصلاح بيئة CI/developer للوصول للسجل.
  - منع الدمج دون مرور `build + test + lint`.

## 4) نقاط قوة بالمشروع
- تقسيم واضح بالموديولات وخدمات مستقلة.
- استخدام DTO و `ValidationPipe` بشكل عام.
- وجود `AllExceptionsFilter` و `LoggingInterceptor` و Correlation ID.
- وجود حماية throttling عامة.
- بنية مناسبة لإضافة observability ومراقبة أفضل بسهولة.

## 5) خطة تحسين مقترحة (Roadmap)

### المرحلة 1: إغلاق المخاطر الحرجة (1-3 أيام)
- تعطيل `synchronize` في كل البيئات غير المحلية.
- إزالة fallback `defaultSecretKey`.
- إصلاح تفويض الطلبات (قراءة/تحديث) حسب الملكية والدور.
- تقييد مسارات المستخدمين الإدارية بدور `ADMIN` فقط.

### المرحلة 2: سلامة البيانات والعمليات (2-4 أيام)
- إصلاح منطق السلة (التحقق التراكمي + cartItemId).
- خصم المخزون أثناء إنشاء الطلب ضمن transaction مع locking.
- إضافة فهارس/قيود مناسبة (مثل uniqueness على عناصر السلة بحسب cart+product+pharmacy).
- إضافة migrations فعلية.

### المرحلة 3: الأمان والتشغيل (2-3 أيام)
- تقييد upload (حجم/نوع/مصادقة).
- توحيد env وإزالة المتغيرات غير المستخدمة.
- تحسين logging (structured logs + no mojibake + تقليل `console.*` المباشر).

### المرحلة 4: الجودة والاختبارات (3-5 أيام)
- إنشاء حزمة اختبارات وحدة وتكامل لسيناريوهات الأعمال الحرجة.
- ضبط CI: `lint`, `build`, `test`, `test:e2e`.
- قياس تغطية مستهدفة أولية (مثلا 60% للخدمات الحرجة).

## 6) توصيات هندسية إضافية
- اعتماد `Request` typing بدل `any` في `@Req()`.
- توحيد صيغة الاستجابات: إما الاعتماد الكامل على `ResponseInterceptor` أو إرجاع raw موحد من الخدمات.
- إضافة versioning للـ API (`/v1`) قبل إزالة مسارات legacy.
- إضافة `CORS` مضبوط للبيئات المختلفة.
- إضافة OpenAPI security schemes بوضوح لكل endpoint.

## 7) خلاصة
المشروع قابل للتطوير ويحتوي أساسًا جيدًا، لكنه يحتاج معالجة فورية لمشاكل الصلاحيات والأمان واتساق البيانات قبل أي توسع وظيفي جديد.  
الأولوية القصوى: تفويض الطلبات/المستخدمين + إعدادات الإنتاج + سلامة المخزون.
